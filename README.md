# MACS FILE COMPRESSION PROJECT
# File Compression Studio

![Status](https://img.shields.io/badge/Submitted-MACS%20JC%20Project%202-0ea5e9?style=for-the-badge)

A Chrome extension project for compressing and decompressing files directly inside the browser. It supports text, images, audio, and video, shows compression metrics, and verifies rebuild quality using hashes for lossless files and quality/bitrate metrics for lossy files. The project is structured to match the MACS JC Project 2 brief and is designed to be readable, modular, and easy to demonstrate.

## Project Overview

This extension provides a popup-based interface where users can upload a file, automatically detect its type, and run an appropriate compression flow. Text files are compressed using lossless GZIP packaging, PNG uses a lossless image path, JPEG uses lossy quality-controlled re-encoding, and audio/video use browser-side FFmpeg WASM pipelines for lossy re-encoding. After processing, the extension shows original size, compressed size, compression ratio, and space savings percentage. It also supports rebuild verification using SHA-256 for lossless outputs and PSNR / bitrate comparison for lossy outputs.

## Features

- Popup-based file upload interface inside the Chrome extension
- Automatic file type detection for:
  - `.txt`, `.csv`
  - `.png`, `.jpg`, `.jpeg`
  - `.wav`, `.mp3`
  - `.mp4`
- Lossless text compression with custom `.fcs` packaging and GZIP payload
- Lossless PNG repack path and lossy JPEG re-encoding
- Lossy audio re-encoding through FFmpeg WASM
- Lossy MP4 re-encoding through FFmpeg WASM
- Compression metrics displayed after processing:
  - Original file size
  - Compressed file size
  - Compression ratio
  - Space savings percentage
- Download support for compressed and rebuilt/decompressed output
- Rebuild verification:
  - SHA-256 hash comparison for lossless outputs
  - PSNR and/or bitrate comparison for lossy outputs
- Graceful user-facing error handling for unsupported types, large files, and failed operations
- Clean responsive popup UI without horizontal overflow

## Installation

### Load unpacked in Chrome

1. Download or clone this project.
2. Open Chrome and go to `chrome://extensions/`.
3. Enable **Developer mode** in the top-right corner.
4. Click **Load unpacked**.
5. Select the `Chrome Extension` folder.
6. Pin the extension from the Chrome toolbar if needed.
7. Click the extension icon to open the popup.

### Vendor libraries

All required libraries are bundled locally inside `lib/`:

- `pako.min.js` — GZIP for text compression
- `upng.min.js` — Lossless PNG re-encoder
- `lamejs.min.js` — MP3 encoder for audio

No network fetches happen at runtime — the CSP blocks them anyway.

## How to Package into a `.crx` File

1. Open `chrome://extensions/`.
2. Make sure **Developer mode** is enabled.
3. Click **Pack extension**.
4. Choose the project root folder containing `manifest.json`.
5. Chrome will generate:
   - a `.crx` file
   - a `.pem` private key file
6. Keep the `.pem` file private.
7. Submit the `.crx` file and your GitHub repository link.

## How to Use

### Compression flow

1. Open the extension popup.
2. Stay on the **Compress** tab.
3. Choose a supported file.
4. Adjust:
   - **Image quality** for JPEG files
   - **Media bitrate factor** for audio/video files
5. Click **Compress File**.
6. Review the results panel:
   - original size
   - compressed size
   - compression ratio
   - space savings
7. Check the **Verification** section.
8. Download the compressed file and rebuilt/decompressed output if needed.

### Decompression flow

1. Switch to the **Decompress** tab.
2. Upload a file generated by the extension, especially `.fcs` for text.
3. Click **Decompress / Inspect**.
4. Review the restored file info and verification message.
5. Download the output.

### Example use cases

- Compress a `.txt` lab report into `.fcs`, then restore it and verify hash equality.
- Compress a `.jpg` photo at 0.75 quality and check PSNR.
- Re-encode a `.wav` file into MP3 and compare bitrate.
- Re-encode an `.mp4` clip with a reduced target bitrate for smaller upload size.

## Rebuild Verification

### Lossless files

For text files and the lossless PNG path, the extension computes a SHA-256 hash of the original content and compares it with the rebuilt output. If both hashes match, the rebuild is an exact byte-for-byte reconstruction.

Displayed fields include:

- Original SHA-256 hash
- Rebuilt SHA-256 hash
- Exact match: Yes / No

### Lossy files

For JPEG images, the extension calculates PSNR to estimate visual similarity. For audio and video, the extension shows bitrate comparison and describes that the rebuild is a lossy approximation rather than an exact original reconstruction.

## Algorithms Used and Justification

### 1. Text: GZIP / DEFLATE (`pako`)

- **Type:** Lossless
- **Why used:** Text is highly redundant and compresses well with dictionary-based methods.
- **Reasonable because:** It preserves exact content, making SHA-256 hash verification meaningful.

### 2. PNG: Lossless image path (`UPNG.js`)

- **Type:** Lossless
- **Why used:** PNG is already a lossless format. UPNG.js re-runs the filter/compression selection more aggressively than most encoders, shaving 10–25 % off most PNGs.
- **Reasonable because:** It avoids any visual degradation.

### 3. JPEG: Canvas-based re-encoding

- **Type:** Lossy
- **Why used:** JPEG is designed for photographs and supports adjustable quality.
- **Reasonable because:** The browser's built-in JPEG encoder is fast and universally available.

### 4. Audio (WAV → MP3 and MP3 → MP3): `lamejs`

- **Type:** Lossy
- **Why used:** WAV is uncompressed; existing MP3s are often at a higher bitrate than needed.
- **How it works:** For both WAV and MP3 inputs, the Web Audio API's `decodeAudioData` converts the file to 32-bit float PCM. `lamejs` then re-encodes it at a target bitrate that is a fraction of the source's effective bitrate (set by the slider, capped at 80 % of the original so the output is always smaller). The target snaps to a standard MP3 bitrate (64, 80, 96, 112, 128, 160, 192, 224, 256, 320 kbps).
- **Reasonable because:** This produces real size reductions instead of leaving MP3 inputs untouched.

### 5. Video (MP4 → WebM): `MediaRecorder` API

- **Type:** Lossy
- **Why used:** `ffmpeg.wasm` is ~30 MB and requires cross-origin isolation headers that are awkward in a Chrome extension popup.
- **How it works:** A hidden `<video>` element plays the source file. Its `captureStream()` is fed to `MediaRecorder` configured for VP9 (with VP8 fallback) + Opus audio, at a target bitrate equal to the slider factor times the source's effective bitrate (minimum 250 kbps). The encode runs at real-time playback speed.
- **Caveat:** Chrome's `MediaRecorder` cannot emit an MP4 container, so the compressed output is `.webm`. This is noted in the verification panel.
- **Reasonable because:** It uses the browser's installed hardware codecs, keeps the extension bundle small, and produces a widely playable file.

## Limitations

- Video output is **WebM**, not MP4. Chrome's `MediaRecorder` API cannot emit MP4 containers. The WebM file is playable in any modern browser and most media players.
- Video re-encoding runs at real-time playback speed (a 60-second clip takes ~60 seconds to process) because `MediaRecorder` is stream-based.
- Very large files can consume significant memory in the popup; the extension caps input at 100 MB.
- Lossless PNG rebuild hash can differ if metadata changes, even when the image still looks identical.
- PSNR is implemented for images, but SSIM is not included in this lightweight build.
- Audio/video "decompression" for lossy media means output inspection or re-download, not reversal to the exact original bytes.
- Chrome extension CSP rules are strict; all vendor libraries are bundled locally inside `lib/` so nothing is fetched at runtime.
